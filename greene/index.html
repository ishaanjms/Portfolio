Designing Intelligence, not just a chatbot

My thesis investigated how designers can work with AI systems that do not behave like traditional deterministic products. Greene's Counsel became the working prototype for that research: an AI-assisted decision-support system built around controlled non-determinism, contextual continuity, and a clear interpretive lens.

01

The Gap

Traditional product design methods assume predictable flows. AI systems introduce probabilistic outputs, evolving behavior, opaque reasoning, and ethical ambiguity.

Behavioural gap

Outputs vary.

The same input can lead to different responses, making consistency and user trust harder to design.

Relational gap

Trust is fragile.

Users need clarity about how the system interprets their context and why it responds in a specific way.

Temporal gap

Systems evolve.

AI products change through model updates, user history, and new data, so design must account for time.

03

Product Direction

I used this framework to explore a more human problem: how people navigate social dynamics, workplace politics, persuasion, conflict, and moments where the “right” answer is rarely obvious.

Human Context

Social situations are messy

People often need help interpreting motives, incentives, power dynamics, and emotional signals before they can decide how to act.

Concept

What if counsel felt direct?

The product imagines a strategic advisory experience inspired by Robert Greene's body of work, as if a seasoned thinker could help frame the situation in real time.

Prototype Goal

Test if AI could become useful counsel

I wanted to see whether an AI product could move beyond generic chatbot responses and provide sharper, contextual, and reflective guidance for complex human decisions.

04

Framework Application

The framework gave the project a clear direction: it helped me decide what the AI should be, how it should behave, and where its boundaries should sit before turning it into a working product.

01 · Foundation

Make it buildable.

I chose a chatbot format and the stack mentioned above so the idea could become a functional counsel system instead of staying abstract.

02 · Purpose

Set the right intent.

The system was framed for betterment: avoiding harmful advice, handling abnormal language carefully, and keeping counsel constructive.

03 · Connection

Keep it conversational.

The product stayed primarily text-based so users could describe messy human situations naturally and receive focused guidance.

04 · Behaviour

Design how it responds.

This shaped the UX: tone, depth, response modes, and features that let the system react differently across different situations.

05 · Evolution

Let context improve it.

Session memory helped the system understand previous context, refine later responses, and evolve toward better results.

05

System Design

Greene's Counsel is structured as three connected layers: context acquisition, framework-driven interpretation, and structured response generation.

Input Layer

Structured context

Situation, tone, depth mode, and conversation history prevent the system from treating complex problems as isolated prompts.

Processing Layer

Persona reasoning

Prompt engineering and a consistent philosophical persona create coherence across changing outputs.

Output Layer

Layered counsel

Responses combine strategic advice, knowledge summaries, and contextual relevance mapping.
